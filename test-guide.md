# TESTING_RULESET_v1 (LLM-Optimized)

> 목적: 생성되는 테스트 코드를 "형식"이 아니라 "회귀 보호막(regression
> safety net)"으로 만들기 위한 규칙 집합\
> 적용 대상: 모든 신규 기능 및 리팩토링 코드\
> 이 문서는 LLM이 테스트를 생성할 때 반드시 따라야 하는 규칙을 정의한다.

------------------------------------------------------------------------

# 1. 필수 구조 규칙

## RULE-1: Arrange → Act → Assert 구조 사용

모든 테스트는 아래 3단계를 명확히 구분한다.

``` ts
// Arrange
// Act
// Assert
```

-   Arrange: 입력/상태 준비
-   Act: 대상 실행
-   Assert: 기대 결과 검증

구조가 명확하지 않으면 수정하라.

------------------------------------------------------------------------

## RULE-2: 테스트 이름은 동작을 설명해야 한다

형식:

    [조건]일 때 → [기대 결과]를 반환한다

금지 예시:

-   works
-   test
-   should work
-   generic naming

테스트 이름만 읽어도 검증 의도가 드러나야 한다.

------------------------------------------------------------------------

## RULE-3: 한 테스트는 한 동작만 검증

-   여러 책임 검증 금지
-   실패 원인이 즉시 드러나야 한다
-   필요 시 테스트를 분리하라

------------------------------------------------------------------------

# 2. 절대 금지 규칙

## RULE-4: 구현 로직 복사 금지

-   기대값 계산식을 구현과 동일하게 작성하지 마라
-   내부 알고리즘을 테스트 코드에서 재현하지 마라
-   명시적 기대값을 사용하라

------------------------------------------------------------------------

## RULE-5: Unit Test에서 외부 의존성 직접 호출 금지

단위 테스트에서 금지:

-   실제 HTTP 요청
-   실제 DB 호출
-   실제 파일 시스템 접근
-   실제 클라우드 API 호출

대신 사용:

-   mock
-   stub
-   fake
-   spy

테스트는 빠르고 반복 가능해야 한다.

------------------------------------------------------------------------

## RULE-6: Integration Test에서 운영 환경 호출 금지

허용:

-   로컬 테스트 DB
-   Docker 컨테이너 DB
-   Fake server
-   Sandbox 환경

금지:

-   Production API
-   운영 DB
-   실제 사용자 데이터

------------------------------------------------------------------------

## RULE-7: 테스트 순서 의존 금지

-   테스트는 독립 실행 가능해야 한다
-   이전 테스트 상태 공유 금지
-   전역 상태 변경 시 반드시 정리(cleanup)

------------------------------------------------------------------------

## RULE-8: Assert 없는 테스트 금지

-   모든 테스트는 최소 1개 이상의 assert 포함
-   try/catch 후 검증 없이 종료 금지
-   예외를 삼켜서 항상 통과하는 구조 금지

------------------------------------------------------------------------

# 3. 기능별 최소 테스트 세트 생성 규칙

모든 신규 기능은 최소 다음 테스트를 생성하라:

## REQUIRED_CASES

1.  정상 경로
2.  오류 경로
3.  경계값

------------------------------------------------------------------------

## 경계값 예시

-   빈 문자열
-   null / undefined
-   최소/최대 숫자
-   최대 길이 문자열
-   배열 길이 0
-   배열 길이 1

------------------------------------------------------------------------

# 4. 조건부 확장 규칙

아래 조건이 코드에 존재하면 반드시 해당 테스트를 추가하라.

------------------------------------------------------------------------

## AUTHORIZATION 존재 시

추가 테스트:

-   권한 부족 시 거부 응답
-   인증 토큰 누락 시 실패
-   만료 토큰 처리

------------------------------------------------------------------------

## CONFIG / ENV 의존성 존재 시

추가 테스트:

-   필수 환경 변수 누락 시 안전 실패
-   잘못된 설정 값 처리

------------------------------------------------------------------------

## 상태 전이 로직 존재 시

추가 테스트:

-   허용된 상태 전이 성공
-   허용되지 않은 상태 전이 차단

------------------------------------------------------------------------

## CACHE 사용 시

추가 테스트:

-   cache hit
-   cache miss
-   TTL 만료

------------------------------------------------------------------------

## 동시성 가능성 존재 시

추가 테스트:

-   중복 요청 처리
-   멱등성 보장
-   race condition 방지

------------------------------------------------------------------------

## 외부 API 호출 로직 존재 시

mock 기반으로 다음 테스트 추가:

-   성공 응답 처리
-   오류 응답 처리
-   timeout 처리
-   rate limit 처리
-   재시도 로직 검증

------------------------------------------------------------------------

# 5. 테스트 생성 후 자동 점검 루틴

테스트 생성 후 아래를 점검하라:

## CHECKLIST

-   [ ] 테스트 이름이 조건+결과 구조인가
-   [ ] Arrange/Act/Assert 구분이 명확한가
-   [ ] 정상/오류/경계 테스트가 존재하는가
-   [ ] 외부 의존성이 mock 처리되었는가
-   [ ] 순서를 바꿔도 통과 가능한 구조인가
-   [ ] assert가 최소 1개 이상 존재하는가
-   [ ] 구현 로직 복사 흔적이 없는가
-   [ ] 테스트가 deterministic한가 (flaky 금지)

조건을 만족하지 않으면 수정하라.

------------------------------------------------------------------------

# 6. 테스트 생성 우선순위 전략

우선 생성 순서:

1.  정상 경로
2.  가장 치명적인 실패 경로
3.  경계값
4.  권한/보안
5.  동시성
6.  외부 시스템 실패

------------------------------------------------------------------------

# 7. 출력 형식 규칙 (LLM 전용)

-   테스트 코드만 출력
-   불필요한 설명 금지
-   TODO 금지
-   Arrange/Act/Assert 구분용 주석만 허용
-   mock은 명시적으로 선언
-   deterministic 보장

------------------------------------------------------------------------

# 8. 핵심 철학

-   테스트는 통과용 코드가 아니다.
-   테스트는 회귀를 막는 보호막이다.
-   빠르고 안정적이며 반복 가능해야 한다.
-   flaky 테스트는 허용하지 않는다.
-   테스트 신뢰성이 최우선이다.
