# Implementer Rules v1

## 목적
- 구현자가 작업 중 즉시 판단 가능한 규칙만 제공한다.
- 모호한 선언("점검했다") 대신 diff로 위반 여부를 확인 가능해야 한다.

## 적용 대상
- `builder/src/**`, `builder/*.html`, `builder/styles.css`, `tests/**`, `scripts/**` 변경 작업
- 문서-only 작업은 일부 규칙을 축약할 수 있다.

## RQ/범위 규칙
1. 모든 구현 작업은 `RQ-xxx`를 기준으로 진행한다.
2. `plan.md`의 target files 밖 수정은 금지한다.
3. 범위 밖 수정이 필요하면 구현을 멈추고 planner/orchestrator 승인을 다시 받는다.

## 파일/함수 크기 규칙
1. 신규 소스 파일은 700줄 초과 금지.
2. 신규 함수는 80줄 초과 금지.
3. 기존 대형 파일 수정 시:
   - 줄 수를 유지하거나 감소시키는 방향을 우선한다.
   - 새 로직은 가능한 한 별도 모듈로 분리한다.

## 모듈 경계 규칙
1. 상태 계산 로직은 `app-state.mjs` 또는 하위 pure helper로 유지한다.
2. DOM 렌더링 책임은 view/render 모듈에 둔다.
3. LLM 호출/정규화는 `llm-*` 계열 모듈에 둔다.
4. 품질 판정 로직은 `quality-gate.mjs`로 집중한다.

## 테스트 규칙
1. 기능 변경 시 테스트를 반드시 갱신한다.
2. 최소 커버리지 단위:
   - 정상 경로 1개
   - 오류 경로 1개
   - 경계값 1개
3. 테스트 작성 스타일은 `test-guide.md`를 따른다.
4. 테스트에서 구현 로직을 복붙해 기대값을 계산하지 않는다.

## 보안/안정성 규칙
1. 비밀값(API key, token, secret) 하드코딩 금지.
2. 사용자 입력/LLM 출력은 정규화 및 기본 검증을 거친다.
3. 외부 호출 실패 시 fallback 또는 안전 실패 경로를 제공한다.
4. 에러 메시지는 원인 파악 가능하지만 민감정보는 포함하지 않는다.

## 금지 패턴
1. 근거 없는 "PASS" 선언
2. 미사용 디버그 코드/주석 잔존
3. 계획 외 파일 변경 후 무보고
4. 테스트 미갱신 상태로 기능 변경 완료 처리

## 완료 보고 규칙
구현자 출력에는 반드시 포함한다.
1. 변경 파일 목록
2. `rq_covered` / `rq_missing`
3. 실행한 테스트/검증 커맨드
4. 잔여 리스크 또는 후속 작업
